# SignalForge Demo Polish

## When To Use

Use this skill when future work touches demo flow, seed data, reset behavior, README polish, screenshots, portfolio presentation, or interview walkthroughs.

## Synthetic Seed Data

Use only synthetic founder/customer/design-partner notes. Make sample data realistic enough to demonstrate discovery synthesis, but avoid real customer, Gitwit, Mechro, interview, recruiter, or private information.

## Demo Reset

Future demo reset behavior should restore a predictable scenario quickly. Keep reset flows obvious, safe, and local-demo friendly.

## README And Screenshots

The README should eventually explain the problem, demo flow, stack, AI abstraction, mock-provider path, and how to run the app. Screenshots should show the actual product workflow, not placeholder marketing sections.

## Interview Demo Flow

Optimize for a short walkthrough: paste or load messy notes, run extraction, review evidence-backed insights, inspect recommendations, and explain next product experiments.

## Gitwit Talking Points

Frame the project as a Gitwit-aligned artifact: product-minded, AI-assisted, pragmatic, demoable, and built in small slices. Keep claims grounded in what the app actually does.
